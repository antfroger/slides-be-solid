What does SOLID means?

  • Single Responsability Principle

SOLID

Single Responsability Principle*

There should never be more than one reason
for a class to change.

* also known as SRP

One responsability per class

  • One reason to change
  • Stay focused on a single concern
  • Simplify maintenance
  • Simplify testing
  • Can be reused (DRY )

One class per responsability

  • Always try to give every class its own responsibility
  • Define for every responsibility a class
  • When the responsability changes, only one class has to be modified

Tips 1/2

If your class name follows one of these rule,
you're probably on the wrong way

  • it contains a separator like And or Or
    The class has probably more than one goal
  • it uses very generic words like Manager, Handler, Helper etc.
    Be careful that the class responsibility is not too large (I won't be so strict about that, but it's maybe vert personnal : it's about semanitc. If, inside ypour team, you define a Manager as a class that access data from one entity, that's fine ; also, we the project grow, having Helpers may help. Lots of big & successful open-source projects are using that kind of classes. It's just a way of being more pragmatic :) )
